# Spec: Auth Email Login Link

## Summary
회사 이메일 로그인 링크 인증, verify entry, 재전송 정책, 인증 실패 처리 기준을 정의한다. 이 문서는 인증 기능 요구사항과 검증 기준의 source of truth다.

## Scope
- 허용 도메인 검증
- 로그인 링크 발급
- 로그인 링크 검증
- 인증 실패 처리
- 요청/검증 비동기 상태
- 최초 로그인 후 이름 수집
- 이름 저장 비동기 상태
- 세션 생성 및 복원
- 로그인 링크 재전송 정책

## Functional Requirements
- 허용 도메인은 `@nurimedia.co.kr` 하나다.
- 운영자가 환경변수로 지정한 bypass 이메일은 로그인 링크 검증 없이 바로 로그인할 수 있다.
- 허용 도메인이 아니면 로그인 링크를 발급하지 않는다.
- 허용 도메인이 아닌 이메일 입력은 로그인 화면에서 처리하고, 입력값은 유지한다.
- 로그인 이메일에는 로그인 전용 URL을 포함한다.
- canonical verify path는 `/auth/verify`다.
- canonical verify query는 `email`, `nonce`를 포함한다.
- migration 동안 client는 legacy root verify query(`/?auth_mode=verify&email=...&nonce=...`)도 해석할 수 있어야 한다.
- 로그인 전용 URL은 직접 로그인에 사용할 수 있다.
- 로그인 전용 URL 유효시간은 5분이다.
- 새 로그인 링크를 발급하면 이전 링크는 무효화된다.
- 로그인 전용 URL은 한 번 사용하면 다시 사용할 수 없다.
- 동일 이메일은 한 cooldown cycle 안에서 최대 5회까지 대기 시간 없이 재전송할 수 있다.
- 동일 이메일의 6번째 요청부터는 5분 cooldown을 적용한다.
- cooldown이 끝나면 해당 이메일의 resend burst count는 reset된다.
- cooldown에 걸린 요청은 로그인 화면에서 처리하고, 남은 대기 시간을 사람이 이해하기 쉬운 형식으로 보여준다.
- 로그인 링크 요청 성공 후에는 페이지 이동 없이 현재 로그인 흐름 안에서 발송 완료 상태를 표시한다.
- 로그인 링크 요청 성공 상태에는 사용자가 입력한 이메일 주소와 재전송 CTA가 함께 보여야 한다.
- 만료된 링크, 이미 사용한 링크, 새 링크 발급으로 무효화된 링크는 인증 실패 화면으로 처리한다.
- 인증 실패 화면에는 다시 링크를 요청하는 CTA와 이메일을 다시 입력하는 CTA가 있어야 한다.
- 실패 이유는 사용자가 이해할 수 있는 설명으로 보여주고, 내부 reason code는 그대로 노출하지 않는다.
- 인증 실패 상태에서는 앱 메인 화면으로 진입할 수 없다.
- 인증 관련 오류는 브라우저 기본 `alert`로 표시하지 않는다.
- 로그인 화면의 요청 상태는 `auth_request = idle | submitting | error`로 관리한다.
- 로그인 링크 진입 검증 상태는 `auth_link_verify = idle | verifying | error`로 관리한다.
- `auth_request = submitting` 동안 로그인 링크 요청 버튼을 다시 누를 수 없다.
- `auth_link_verify = verifying` 동안 인증 결과가 확정되기 전까지 앱 메인 화면으로 진입할 수 없다.
- request-link 또는 verify-link 요청이 지연되거나 응답하지 않아도 auth flow는 무한 대기 상태에 머물지 않고 terminal auth state로 수렴해야 한다.
- 로그인 링크 진입 URL에서 새로고침하거나 stale verify query 상태로 다시 진입해도 검증은 무한 대기 상태에 머물지 않고 terminal auth state로 수렴해야 한다.
- 이미 유효한 세션이 있는 상태에서 stale verify query로 다시 진입하면 verify를 반복하지 않고 세션 복원 경로를 우선 적용한다.
- 로그인 화면은 모바일과 데스크톱 모두 같은 auth surface hierarchy를 사용한다.
- 로그인 화면에는 브랜드 식별 요소와 로그인 목적 안내가 있어야 한다.
- 이메일 input은 접근성용 label을 유지해야 한다.
- 이메일이 비어 있거나 형식이 유효하지 않으면 로그인 링크 요청 버튼은 비활성화된다.
- 로그인 이메일은 서비스 식별, 로그인 링크, 만료 안내, 대체 진입 수단을 포함해야 한다.
- 로그인 성공 시 같은 브라우저에서 최대 90일 세션을 유지한다.
- bypass 이메일 목록 자체는 환경변수로만 관리하고 tracked source에는 직접 적지 않는다.
- 비로그인 사용자는 보호된 화면과 API에 접근할 수 없다.
- 로그인 성공 후 사용자 이름이 비어 있으면 이름 입력 화면으로 보낸다.
- 이름 입력은 단일 input field 하나로 받는다.
- 이름 input은 비울 수 없다.
- 이름은 1글자 이상 입력되면 유효하다.
- 이름은 최대 10글자까지 입력할 수 있다.
- 10글자를 초과하는 타이핑은 입력되지 않아야 하고, 붙여넣기된 초과분은 제거한다.
- 이름 저장 상태는 `name_submit = idle | submitting | error`로 관리한다.
- `name_submit = submitting` 동안 저장 버튼을 다시 누를 수 없다.
- 이름 저장 실패 시 이름 입력 화면에 머물며 입력값을 유지한다.
- 이름 입력 완료 전에는 앱 메인 화면으로 진입할 수 없다.
- 이름이 이미 있는 사용자는 이름 입력 화면을 건너뛴다.

## Acceptance Criteria
- 허용 도메인이 아닌 이메일은 로그인 링크를 발급할 수 없다.
- 허용 도메인이 아닌 이메일 입력 시 로그인 화면에 실패 상태가 보이고 입력 이메일이 유지된다.
- 로그인 이메일에는 `/auth/verify` 기반 로그인 전용 URL이 포함된다.
- migration 동안 legacy root verify query도 계속 동작한다.
- 로그인 전용 URL만으로 로그인이 가능하다.
- 로그인 이메일에서 서비스 식별, 로그인 링크, 5분 유효 안내, 대체 진입 수단을 확인할 수 있다.
- 로그인 전용 URL은 5분 뒤 만료된다.
- 만료된 링크로 진입하면 인증 실패 화면이 보이고 로그인되지 않는다.
- 새 로그인 링크를 발급하면 이전 링크는 인증 실패 화면으로 처리된다.
- 같은 이메일은 burst 5회까지 즉시 재요청할 수 있다.
- 같은 이메일의 6번째 요청부터는 5분 cooldown이 적용되고 남은 대기 시간이 사람이 이해하기 쉬운 형식으로 표시되며 입력 이메일이 유지된다.
- 사용한 링크는 다시 사용할 수 없고 인증 실패 화면으로 처리된다.
- 인증 실패 화면에는 재시도 CTA들이 보인다.
- 인증 실패 화면의 `이메일 다시 입력`을 누르면 로그인 화면으로 돌아간다.
- 인증 실패 화면에는 raw reason code가 아니라 사람이 이해 가능한 설명이 보인다.
- 로그인 링크 요청 중에는 요청 버튼을 다시 누를 수 없다.
- 로그인 링크 요청 성공 후 같은 컴포넌트 안에서 입력한 이메일 주소를 포함한 발송 안내가 보인다.
- 로그인 화면은 모바일과 데스크톱 모두 같은 auth surface hierarchy로 표시된다.
- 로그인 화면에 불필요한 보조 설명이 보이지 않는다.
- 유효한 이메일 형식을 입력하기 전까지 로그인 링크 요청 버튼이 비활성화된다.
- 로그인 링크 검증 중에는 처리 중 상태가 보이고 결과 확정 전 앱 메인 화면으로 이동하지 않는다.
- 로그인 링크 검증 중 새로고침하거나 네트워크가 hang되어도 무한히 `verifying`에 머물지 않는다.
- stale verify query와 유효 세션이 함께 있을 때는 세션 복원 후 앱 진입이 가능하다.
- 로그인 성공 시 세션이 생성되고 재접속 시 복구된다.
- 환경변수로 지정된 bypass 이메일은 링크 클릭 없이도 로그인된다.
- 로그인 성공 후 이름이 비어 있으면 이름 입력 화면으로 이동한다.
- 이름 input은 한 개이며, 빈 값은 허용되지 않는다.
- 이름 input은 최대 10글자까지만 유지된다.
- 이름은 1글자 이상이면 저장할 수 있다.
- 이름 저장 중에는 진행 중 상태가 보인다.
- 이름 저장 중에는 저장 버튼을 다시 누를 수 없다.
- 이름 저장 실패 시 입력한 이름이 유지된다.
- 이름 입력을 완료해야 앱 메인 화면으로 진입할 수 있다.
- 이름이 이미 저장된 사용자는 이름 입력 없이 앱으로 진입한다.
