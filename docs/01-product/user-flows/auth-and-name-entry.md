# Auth And Name Entry

## Scope
- 회사 이메일 로그인 링크 요청
- 로그인 링크 검증
- 인증 실패 처리
- 이름 입력 온보딩
- 세션 복구
- 로그인 링크 재전송 정책

## Authentication Flow
1. 사용자가 웹사이트에 접속한다.
2. 로그인 화면에서 회사 이메일을 입력한다.
3. 시스템이 허용 도메인인지 확인한다.
4. 허용 도메인이 아니거나 로그인 링크를 요청할 수 없는 상태면 로그인 화면에 그대로 머물며 실패 이유를 확인한다.
5. 요청이 가능하면 이메일로 로그인 링크가 발송되고, 사용자는 같은 로그인 흐름 안에서 발송 완료 상태를 본다.
6. 사용자가 로그인 링크를 클릭한다.
7. 로그인 링크가 유효하면 시스템은 사용자 이름 존재 여부를 확인한다.
8. 로그인 링크가 만료되었거나, 이미 사용되었거나, 더 새 링크 발급으로 무효화되었으면 사용자는 인증 실패 화면을 본다.
9. 인증 실패 화면에서 다시 링크를 요청하거나 이메일을 다시 입력한다.
10. 이름이 비어 있으면 이름 입력 화면으로 이동한다.
11. 이름 입력이 완료되면 장소 탐색 화면으로 이동한다.
12. 같은 브라우저 재방문 시 저장 세션이 유효하면 로그인 화면을 건너뛴다.
13. 로그인 링크 검증 중 예외가 생겨도 사용자는 무한 대기 상태에 머물지 않고, 다시 시도 가능한 상태로 돌아갈 수 있어야 한다.

## Rules
- 로그인 링크는 직접 로그인에 사용할 수 있어야 한다.
- 로그인 링크는 제한된 시간 동안만 유효해야 한다.
- 새 로그인 링크를 발급하면 이전 미사용 링크는 더 이상 사용할 수 없어야 한다.
- 로그인 링크는 한 번 사용하면 다시 사용할 수 없다.
- 반복 재요청에는 재전송 제한과 cooldown 정책이 적용된다.
- `비허용 도메인`, 재요청 제한 같은 요청 단계 문제는 로그인 화면에서 처리한다.
- 만료/사용 완료/무효화처럼 검증 단계 문제는 인증 실패 화면에서 처리한다.
- 인증 실패 화면에서는 사용자가 앱 메인 화면으로 우회 진입할 수 없어야 한다.
- 저장 세션과 오래된 verify 진입 정보가 동시에 있으면, 사용자는 끊기지 않는 세션 복원 결과를 보게 되어야 한다.
- 인증 실패 이유는 시스템 내부 코드가 아니라 사람이 이해할 수 있는 설명으로 보여야 한다.

## Name Capture Flow
1. 로그인 성공 후 이름이 비어 있는 사용자는 이름 입력 화면을 본다.
2. 이름 입력은 단일 이름 입력 흐름으로 진행한다.
3. 이름은 비워 둘 수 없다.
4. 최소 1글자 이상 입력하면 제출할 수 있다.
5. 저장 성공 후 장소 탐색 화면으로 이동한다.

## Failure Expectations
- 로그인 요청이 실패해도 사용자가 입력한 이메일은 유지한다.
- 인증 실패 화면에서는 로그인 재시도 경로가 항상 보인다.
- 인증 실패 화면은 auth 흐름 바깥으로 이탈하지 않고, 다시 링크 요청 또는 이메일 재입력으로 이어져야 한다.
- 인증 실패 화면에는 raw reason code가 아니라 사람이 이해할 수 있는 설명이 보여야 한다.
- 이름 저장 실패 시 이름 입력 화면에 머물며 입력값을 유지한다.
- 로그인 링크 검증 중 예외나 네트워크 지연이 발생해도 무한 로딩 대신 다시 시도 가능한 terminal auth state가 보여야 한다.
