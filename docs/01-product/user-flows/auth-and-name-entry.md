# Auth And Name Entry

## Scope
- 회사 이메일 로그인 링크 요청
- 로그인 링크 검증
- 인증 실패 처리
- 이름 입력 온보딩
- 세션 복구
- 로그인 링크 재전송 정책

## Authentication Flow
1. 사용자가 웹사이트에 접속한다.
2. 로그인 화면에서 회사 이메일을 입력한다.
3. 시스템이 허용 도메인인지 확인한다.
4. 허용 도메인이 아니거나 로그인 링크를 요청할 수 없는 상태면 로그인 화면에 그대로 머물며 실패 이유를 확인한다.
5. 요청이 가능하면 이메일로 canonical 로그인 전용 URL(`/auth/verify?email=...&nonce=...`)을 발송하고, 페이지 이동 없이 현재 로그인 컴포넌트에서 입력한 이메일 주소와 함께 발송 안내를 보여준다.
6. 사용자가 로그인 전용 URL을 클릭한다.
7. 로그인 전용 URL이 유효하면 사용자 이름 존재 여부를 확인한다.
8. 로그인 전용 URL이 만료되었거나, 이미 사용되었거나, 새 링크 발급으로 무효화되었으면 인증 실패 화면을 본다.
9. 인증 실패 화면에서 `새 로그인 링크 받기` 또는 `이메일 다시 입력`을 선택한다.
10. 이름이 비어 있으면 이름 입력 화면으로 이동한다.
11. 이름 입력이 완료되면 장소 탐색 화면으로 이동한다.
12. 같은 브라우저 재방문 시 저장 세션이 유효하면 로그인 화면을 건너뛴다.
13. 로그인 링크 검증 중, stale verify query 상태, refresh, hard refresh 유사 상황에서도 auth가 무한 대기 상태에 머물지 않는다.

## Rules
- canonical 로그인 전용 URL path는 `/auth/verify`다.
- migration 동안 legacy root verify query(`/?auth_mode=verify&email=...&nonce=...`)도 해석할 수 있다.
- 로그인 전용 URL은 직접 로그인에 사용할 수 있어야 한다.
- 로그인 전용 URL은 5분 동안만 유효하다.
- 새 로그인 링크를 발급하면 이전 로그인 전용 URL은 무효화된다.
- 로그인 전용 URL은 한 번 사용하면 다시 사용할 수 없다.
- 동일 이메일은 burst 5회까지 대기 시간 없이 재전송할 수 있다.
- 동일 이메일의 6번째 요청부터는 5분 cooldown을 적용한다.
- cooldown 메시지는 `MM분 SS초 후에 다시 시도해주세요.` 형식을 사용하고, 분이 0이면 초만 표시한다.
- `비허용 도메인`, `재요청 cooldown`은 로그인 화면에서 처리한다.
- `만료된 링크`, `이미 사용된 링크`, `무효화된 링크`는 인증 실패 화면에서 처리한다.
- 인증 실패 화면에서는 앱 메인 화면으로 진입할 수 없다.
- stale verify query와 유효 세션이 동시에 있으면 세션 복원을 우선한다.
- 인증 실패 이유는 raw code가 아니라 사람이 이해할 수 있는 한국어 문구로 표시한다. generic failure는 `인증에 실패했어요. 새 로그인 링크를 다시 받아주세요.`를 사용한다.

## Name Capture Flow
1. 로그인 성공 후 이름이 비어 있는 사용자는 이름 입력 화면을 본다.
2. 이름 입력은 성과 이름을 포함한 단일 input field 하나로 받는다.
3. 이름 input은 비울 수 없다.
4. 1글자 이상 입력하면 제출할 수 있다.
5. 저장 성공 후 장소 탐색 화면으로 이동한다.

## Failure Expectations
- 로그인 요청이 실패해도 사용자가 입력한 이메일은 유지한다.
- 인증 실패 화면에서는 로그인 재시도 경로가 항상 보인다.
- 인증 실패 화면에는 raw reason code가 아니라 한국어 설명이 보여야 한다.
- 이름 저장 실패 시 이름 입력 화면에 머물며 입력값을 유지한다.
- 로그인 링크 검증 중 예외나 네트워크 지연이 발생해도 무한 로딩 대신 terminal auth state가 보여야 하고, generic failure는 `인증에 실패했어요. 새 로그인 링크를 다시 받아주세요.`를 사용한다.
